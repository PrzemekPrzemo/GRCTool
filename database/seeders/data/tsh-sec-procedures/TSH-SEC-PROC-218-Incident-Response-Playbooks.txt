INCIDENT RESPONSE PLAYBOOKS
BY PRZEMYSŁAW SZULECKI, CSO
Trzy playbooki: ransomware/malware, wyciek credentiali przez GitHub, phishing z przejęciem konta. Każdy: fazy STOP → OCENA → DECYZJE → ODBUDOWA z konkretnymi krokami i narzędziami.
TSH-SEC-PROC-218  |  v1.0  |  2025-01-01  |  CSO — Przemysław Szulecki
Playbooki = akcja, nie teoria. Każdy scenariusz to lista kroków w kolejności. Nie zaczynaj od analizy — zacznij od STOP i CONTAIN. Czytasz to w kryzysie.
Każdy playbook zakłada że znasz już PROC-202 (IR Procedure). Tu znajdziesz tylko kroki specyficzne dla danego scenariusza, w kolejności priorytetów.
# Playbook 1: Ransomware / Malware na urządzeniu
Pierwsze 30 minut decyduje o skali incydentu. Izoluj zanim zbadasz.
## Faza 1: STOP (0–15 minut)
## Faza 2: OCENA ZASIĘGU (15–60 minut)
## Faza 3: DECYZJE (CSO, do 2 godzin)
Powiadomienie klienta? → Tak jeśli urządzenie miało dostęp do danych klienta → Szablon #1
Powiadomienie UODO? → Tak jeśli dane osobowe były na urządzeniu → Szablon #2 (72h)
Powiadomienie Management Board? → Tak dla każdego P1 → Szablon #4 (1h od klasyfikacji)
Wipe urządzenia? → Tylko gdy forensics kompletne lub gdy nie ma potrzeby zachowania dowodów
NIGDY nie płać okupu bez decyzji Management Board i konsultacji z prawnikiem. TSH ma polisę cyber insurance — poinformuj ubezpieczyciela przed jakąkolwiek płatnością.
## Faza 4: ODBUDOWA
Przywróć dostęp z czystego urządzenia — IRU enrollment nowego MacBooka
Zresetuj wszystkie hasła i tokeny, do których użytkownik miał dostęp (sprawdź 1Password)
Przywróć kod z GitHub (nie z zainfekowanego urządzenia)
Monitoruj przez 72h — Entra ID Identity Protection + MDE
# Playbook 2: Credentials / dane wrażliwe w repozytorium GitHub
Każda sekunda liczy. Publiczne repo z kluczem API = klucz już jest skompromitowany — nie ma znaczenia czy go usuniesz. Rotuj natychmiast.
## Faza 1: STOP (0–10 minut)
## Faza 2: OCENA (10–30 minut)
## Faza 3: DECYZJE
Klucz do infrastruktury klienta → Natychmiast powiadom klienta (Szablon #1), nie czekaj na pełną analizę
Klucz do infrastruktury TSH → IT Lead rotuje i monitoru pod kątem nieautoryzowanego użycia
Dane osobowe w repo → ocena GDPR breach — UODO 72h (Szablon #2)
Brak oznak exploit → zamknij jako P2; post-incident review obowiązkowy
## Faza 4: ZAPOBIEGANIE NAWROTOWI
Włącz GitHub Push Protection jeśli jeszcze nie aktywne: GitHub → Org → Code security → Secret scanning → Push protection: ENABLE
Sprawdź czy wszystkie repo mają Branch Protection + secret scanning (CFG-304)
Sesja z zespołem developerskim: jak bezpiecznie zarządzać sekretami (1Password, GitHub Secrets)
# Playbook 3: Phishing — pracownik kliknął / podał dane
## Faza 1: STOP (0–15 minut)
## Faza 2: OCENA (15–60 minut)
## Faza 3: PRZYWRÓCENIE DOSTĘPU
Reset hasła Entra ID (przez CSO/IT Lead, nie przez użytkownika samodzielnie)
Weryfikacja MFA — upewnij się że to TSH resetuje, nie atakujący
Przejrzyj i usuń wszelkie reguły mailowe utworzone przez atakującego (Google Mail → Settings → Filters)
Sprawdź ustawienia email: forwarding, vacation responder, authorized apps
Re-enable konto dopiero gdy IT Lead potwierdza czystość
## Faza 4: DECYZJE I FOLLOW-UP
Dostęp do danych klienta? → Szablon #1 do klienta w 24h
Dane osobowe narażone? → Ocena UODO (Szablon #2)
Użytkownik przeszedł szkolenie z phishingu? → Remedial training obowiązkowy
Aktualizacja phishing simulation template — dodaj podobny scenariusz do następnej raty
