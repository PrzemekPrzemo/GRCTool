During a P1/P2 incident: step-by-step procedure is here. Communication templates: PROC-217. Scenario playbooks: PROC-218. Classification and notification policy: POL-007.
# 1. Quick Reference
# 2. Phase 1: Detect & Report
Anyone who suspects an incident: email security@tsh.io AND call IT Lead immediately
IT Lead: acknowledge within 15 minutes; classify P1–P4; open REG-009 entry
P1/P2: assemble response team (CSO + IT Lead + relevant PM) immediately
Do NOT investigate alone — report first, contain together
# 3. Phase 2: Contain
## 3.1 Device Compromise — Apple (IRU)
IRU Console (tsh.iru.io) → Devices → [device] → Lock — use LOCK not Wipe to preserve forensic evidence
If data protection is priority over forensics: IRU → Wipe
Check: was device used for client project? If yes — client notification clock starts NOW (PROC-217 Template #1)
Revoke all Entra ID sessions: Entra Admin → Users → [user] → Revoke sessions
## 3.2 Account Compromise
Entra ID → Users → [user] → Disable account + Revoke all sessions
PIM: revoke any active privileged role activations
GitHub: remove from org if account believed compromised
Rotate all credentials the account had access to (check 1Password shared vaults)
## 3.3 Ransomware / Malware
IRU Lock (not Wipe) — isolate device while preserving forensics
Check blast radius: what other systems did the affected device/account have access to?
Do NOT pay ransom — escalate to Management Board immediately
For detailed step-by-step playbooks for ransomware, GitHub credential leak, and phishing: see PROC-218 (Incident Response Playbooks).
# 4. Phase 3: Analyse
Collect logs: Entra ID audit, IRU activity log, MDE device timeline, GitHub audit log
Determine: scope of compromise, data accessed, timeline, root cause
Document findings in REG-009 in real time
# 5. Phase 4: Notify
Use PROC-217 (Incident Communication Templates) for all external communications — templates are pre-approved and require only [BRACKET] completion:
# 6. Phase 5: Eradicate & Recover
Remove threat: re-image device via IRU/Intune; rotate all compromised credentials
Restore from clean backup (GitHub for code; Google Drive for documents)
Verify all controls working before returning device/account to service
Monitor for 72h post-recovery: Entra ID Identity Protection + MDE
# 7. Phase 6: Post-Incident Review
P1/P2: mandatory post-incident report within 2 weeks; presented to Management Board
Root cause analysis; lessons learned → update training (PROC-207) and policies
Close REG-009 entry with final details and lessons learned
Speed of containment beats completeness of investigation every time. Contain first, understand later.
