EXCEPTION MANAGEMENT POLICY
BY PRZEMYSŁAW SZULECKI, CSO
Defines the process for requesting, approving, and tracking exceptions to TSH ISMS requirements. All exceptions require CSO approval, have a maximum 12-month duration with no auto-renewal, and are logged in REG-012.
TSH-SEC-POL-017  |  v1.0  |  2026-07-14  |  CSO — Przemysław Szulecki
# 1. Purpose & Scope
This policy defines the process for requesting, approving, documenting, and reviewing exceptions to TSH ISMS requirements. An exception is a documented and approved deviation from a policy or control requirement. Exceptions are not a normal state — every exception must be time-boxed, justified, backed by compensating controls, and approved by CSO. This policy applies to all TSH personnel and all TSH security policies.
# 2. Exception Principles
No exception is permanent — all exceptions have a defined expiry date (maximum 12 months, no auto-renewal)
All exceptions require CSO approval — verbal approvals are not valid
Exceptions are compensating controls, not policy bypasses — the risk must still be managed
Exceptions are auditable — documented in REG-012 (Exception Register)
Zero tolerance exceptions — certain areas cannot be excepted: MFA requirement; encryption of RESTRICTED data; client data handling rules; GDPR breach notification timelines

# 4. Exception Request Process
Submit exception request to CSO via Jira: IT-SECURITY → Security Exception Request
Include: policy affected; specific requirement being excepted; business justification; risk assessment; proposed compensating controls; requested duration
CSO reviews within 3 business days; may request additional information
If approved: CSO documents in REG-012 with approval, conditions, and review date
If rejected: requester notified with reason; may escalate to Management Board
Exception tracked until expiry; IT Lead sends reminder 14 days before expiry
At expiry: exception either remediated (control gap fixed), renewed (with fresh approval), or escalated
# 5. Compensating Controls
Every approved exception must include compensating controls that reduce the residual risk to acceptable level. Examples:
Cannot apply MFA to legacy system → network isolation + additional monitoring
Cannot encrypt data at rest on legacy platform → additional access logging + quarterly access review
Cannot rotate API key immediately → rotate within 48h; add key to monitoring for anomalous use
# 6. Exception Register (REG-012)
IT Lead maintains REG-012 (Security Exception Register). Monthly review by CSO. Quarterly summary to Management Board. All exceptions visible to CSO and Management Board — no hidden exceptions.
If you are working around a security control because it is inconvenient, that is a policy violation — not an exception. Exceptions exist for genuine business or technical constraints. When in doubt, ask CSO.
